Bill Pay v3.5.1

This release note explains the changes to this capability.

New features

The new features in this release are:

  • Bill Pay Ebills Payveris Enrolment widget

    A new widget that allows a Bill Pay customer to enable eBills by enrolling with an electronic payee that supports the service and start receiving their bills electronically.

  • The ability to stop eBills

    A new feature in the Bill Pay Payee List widget that allows a Bill Pay customer to stop receiving their bills electronically from a payee that has eBills enabled.

  • The eBill payment amount

    In the Bill Pay Manage Payments widget, when a customer makes a payment to the payee with an outstanding amount due, the amount due or the minimum amount due are available as the payment option.

Payments v3.5.1

This release note explains the changes to this capability.

New features

The new features in this release are:

  • View submitted stop check requests (web)

    The new Stop Checks List widget allows the bank customer to view a list of submitted requests to stop the execution of checks.

    The widget has the following features:

    • View a list of submitted stop checks.

    • Click on a specific stop check and view the details.

    • Print the details of a stop check request.

  • Activate external accounts for A2A external transfers (web)

    Account-to-account external transfers are quick and easy money transfers between accounts that the customer has access to in different banks. The external US account has to be linked and activated in order to initiate A2A external transfers. Payments now supports both connecting and activation of an external US account. The functionality to initiate an A2A transfer has not been released yet. When a customer connects an external US account, two micro deposits are credited to that account. Micro deposits are amounts less than $1.

    The status of the request then changes to PendingActivation. As part of the activation process, the customer has to correctly input these micro deposits and thereby successfully activate an account. In case of incorrect micro deposit amounts, the customer is notified of an error.

    Upon successful activation, Payments makes the external US account available for external A2A transfers.

    Note
    		 This feature is in BETA release. It is not supported for use in production and may be subject to unannounced breaking changes.

Bug fixes

The following issues have been fixed in this release:

  • For iOS, it was not possible to set the ID in the BBSEPAFormSpec class. It was only validating IBAN schemes. Now it is possible to set the ID.

Known issues

The known issues in this release are:

  • Introduced in v3.5.0

    • PAYM-135: While activating an external account for A2A transfers, the user can tab across the disabled fields and see a blinking cursor.

  2. Deprecated Products widgets and client

    The Product Summary client and widget and the Accounts client are officially deprecated and they will not be released from version 3.7.0 onwards.

    The Products Overview widget, the Product Details widget, and the Products client have been introduced as a replacement of the previously mentioned deprecated clients and widget. The Products client and widget are available starting from Widget Collection Retail Banking version 3.5.0.

    Follow this guide to update to the latest widget collection.

    • Android Guidelines

      The Products client is a reconstructed version of the Product Summary client. It provides a better DBS alignment and better extendability of the Recipient Contract.

      The following changes have been made:

      From To

      ProductSummaryClient

      ProductsClient

      ProductSummaryContract

      ProductsOverviewContract

      AggregatedBalance (from product-summary-client)

      AggregatedBalance (from products-client)

      Product

      BaseProduct (and specific Product Type classes. i.e. CreditCard, CurrentAccount, Loan, etc.)

      ProductKind (from product-summary-client)

      ProductKind (from products-client) interface is implemented by specific classes i.e. CreditCards, CurrentAccounts, Loans, etc.

      ProductKinds

      Java: ProductUtils.getProductKinds(productSummary)

      Kotlin: productSummary.getProductKinds()

      load method → returns an instance of ProductKinds class

      getProductSummary method → returns an instance of ProductSummary class

      getProductKinds method → returns an instance of ProductKinds class

      getProductSummary method → returns an instance of ProductSummary class

      getRecipients converts an instance of ProductKinds class to RecipientModel list privately

      getRecipients converts an instance of ProductSummary class to RecipientModel list by using ProductConverterFactory class

      1. Migrate to Products client

        Migrate from using the Product Summary client and/or Accounts client to using the Products client by following the steps below.

        1. Add the Products client dependency in Gradle

          implementation com.backbase.android.clients:products-client:3.5.0

        2. Replace all Product Summary client configurations

          Replace the client name:

          ProductSummaryClient

          with

          ProductsClient

        3. Use ProductSummaryData from the Products client instead

          1. Replace interface

            If you are extending ProductSummaryData from product-summary-client, then replace it with ProductSummaryData from products-client.

          2. Retrieving products

            In order to be able to retrieve product summary, use the getProductSummary method. This method is the replacement method for the load and getProductKinds methods from the Product Summary client.

            In the Product Summary client, the response was returning an instance of the ProductKinds class. In the Products client, the response is fully aligned with DBS; therefore, the response returns an instance of the ProductSummary class. The ProductSummary class has some Kotlin extension functions, such as getProductKinds, isEmpty, and getProductsList. To use it from Java, use ProductUtils.

            Example usage:

            val params = ProductSummaryParamsBuilder()
            .creditAccount(true)
            .debitAccount(true)
            .maskIndicator(true)

client.getProductSummary(object : ProductSummaryListener {
    override fun onSuccess(productSummary: ProductSummary) {
        // handle the response
    }

    override fun onError(errorResponse: Response) {
        // handle the response
    }
}, params)

        4. Extend RecipientContract

          Previously the only way to extend the recipient contract implementation was subclassing the client and changing the existing implementation of the getRecipients method which is implemented as part of RecipientContract.

          In the Products client, a new way of customising the recipient contract is introduced. For more information, check Customize Product Converters.

          Also, the Products client is not filtering by the 'external transfer allowed flag'; therefore, you need to extend the Selector widget in order to be able to use this filter. See Selector widget for more details.

        5. Migrate from the Accounts client to the Products client

          There are not many things changed during the transition from the Accounts client to the Products client, but now a better DBS alignment is provided. This means the following:

          • Deprecated fields in the Accounts client are not present in the Products client.

          • The`ArrangementItemBase` class is renamed to ArrangementBaseItem in the Products client.

          • The newly added properties in DBS are available in the Products client.

          • The Date object types are changed to the String type, added convenience getter methods for parsing to Date objects.

      2. Migrate from the Product Summary widget to the Products widget

        In the Products widget, every view is separated into a sub-widget which is inherited from a base Products widget. The base Products widget is present to enable shared functionality between the sub-widgets, for instance storing pub/sub event keys that are shared.

        These sub-widgets are called:

        • ProductsOverviewWidget

        • ProductDetailsWidget

    • iOS Guidelines

      1. The Products client is a reconstructed version of the Product Summary client. It provides a better DBS alignment and better extendability of the Recipient Contract.

        The following changes have been made:

        From To

        ProductSummaryClient

        ProductsClient

        ProductSummaryClientProtocol

        ProductSummaryProtocol

        AggregatedBalance

        BBAggregatedBalance

        Product

        BBBaseProduct & Specific Product Type classes. eg. BBCurrentAccount

        ProductKind

        BBProductKind protocol is implemented by Specific Product’s array type classes. eg. BBCurrentAccounts

        ProductKinds

        productSummary.productKinds()

        ProductFormatter

        BBProductFormatter

        load method → returns an instance of ProductKinds class

        retrieveProductSummary method → returns an instance of BBProductSummary class

        retrieveProductKinds method → returns an instance of ProductKinds class

        retrieveProductSummary method → returns an instance of BBProductSummary class

        getRecipients converts an instance of ProductKinds class to RecipientModel list privately

        getRecipients converts an instance of BBProductSummary class to RecipientModel list by using BBProductConverterFactory class.

      2. Migrate to Products client

        Migrate from using the Product Summary client and/or Accounts client to using the Products client.

        1. Add Products client pod

          pod 'ProductsClient', '5.0.0' /*or later*/

        2. Replace all Product Summary client configurations

          Replace the client name:

          ProductSummaryClient

          declarations with

          ProductsClient

        3. Migrate from ProductSummaryClientProtocol to ProductSummaryProtocol

          1. Replace protocols

            If you are extending ProductSummaryClientProtocol, then replace it with the new ProductSummaryProtocol.

          2. Retrieving products

            In order to be able to retrieve Product Summary, use the retrieveProductSummary method. This method is the replacement method for the load and retrieveProductKinds methods from the Product Summary client.

            In the Product Summary client, the response was returning an instance of the ProductKinds class. In the Products client, the response is fully aligned with DBS; therefore, the response returns an instance of the BBProductSummary class. In the BBProductSummary class, there is a helper getter method called productKinds. This getter method returns the list of product kinds that you need.

            Example usage:

            let params = BBProductSummaryParams(debitAccount: nil, creditAccount: nil, maskIndicator: nil)

client.retrieveProductSummary(with: params, completionHandler: { (productSummary, error) in
    print(productSummary.productKinds().description)
})

        4. Extend RecipientContract

          Previously the only way to extend the recipient contract implementation was subclassing the client and changing the existing implementation of the getRecipients method which is implemented as part of RecipientContract.

          In the Products client, a new way of customising the recipient contract is introduced. For more information, see Customize Product Converters.

          The Products client is not filtering by the external transfer allowed flag; therefore, you need to extend the Selector widget in order to be able use this as a filter. See Selector widget for more details.

        5. Migrate from the Accounts client to the Products client

          There are not many changes needed for the transition from the Accounts client to the Products client. However, to improve alignment with DBS, the following is needed:

          • Deprecated fields in the Accounts client are not present in the Products client.

          • The`ArrangementItemBase` class is renamed to ArrangementBaseItem in the Products client.

          • The newly added properties in DBS are available in the Products client.

          • The NSDate object types are changed to NSString type, added convenience getter methods for parsing to NSDate objects.
