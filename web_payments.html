Web widgets

Understand Payments describes the functionalities of Payments. The following web widgets deliver these functionalities:

  • Initiate payment

    • Initiate omni payment: Dynamic payment widget that you can easily configure for various payment types. It comes with out-of-the-box configurations for internal transfer and A2A external transfer.

    • Quick transfer: Transfer funds to internal, connected, or a contact’s US account with minimum fields.

    • Initiate payment wizard: A single universal interface that guides the bank account user through the initiate payment process. Supports SEPA credit transfer, Faster Payments, CHAPS, and international transfers.

  • Connect an account for A2A external transfer

    • Connect external account: Connect an external account that you hold in a different bank.

  • Manage payment orders

    • Manage upcoming payments: View all future dated payments in a list.

  • Stop checks

    • Request stop checks: Submit a request to stop the clearance of an issued or lost check.

    • View stop check payment requests: View submitted stop check requests.

Initiate payment wizard

The Initiate payment wizard uses is a UI that guides the account user through creating a payment in any payment type the bank supports.

In contrast to the omni-payment widget solution where the account user can define separate entry points for each payment type, the Initiate payment wizard widget offers a single unified UI that guides the user to the correct payment types based on the initial inputs. The widget also provides possible delivery options for each possible payment types such as instruction priority, transfer fee, fee sharing options, the execution timeframe, and the cutoff time.

This helps the user to make an informed choice while maintaining the control over the payment method. You can re-use the omni payment configurations for the supported payment types or methods to power the payment wizard widget. Payment types supported by the Initiate payment wizard widget are SEPA CT, Faster Payments, CHAPS and International transfers.

When the account user opens the payment wizard, they will see the following fields:

  • From account: Required, select an account.

    • Select shows all accounts the user has access to and are allowed to debit, across all supported payment types.

  • Amount and currency: Required, choose currency and a enter an amount to pay.

    • Set to default currency set up by the bank.

    • Change currency to a different available currency.

      • Most popular currencies displayed at the top of the list.

      • Displays country three letter ISO code and name of currency.

      • Search for a currency by its ISO code and name.

  • Transfer to

    • Beneficiary’s name: Required, select a saved beneficiary or enter a new one.

      • Shows all accounts the user has access to and are allowed to credit, except the account currently selected, in the From account field.

      • Shows all beneficiary accounts listed in users contacts.

      • Auto-populates beneficiary name, account, bank country and bank branch code, if available.

  • Country: Required, appears when initiating an international transfer.

    • Shows by default the beneficiary bank’s home country.

    • Change the country to another available country.

    • Displays the name and national flag of the country, and the country’s ISO code.

    • Search for a country by its ISO code and name.

    • Changing the country shall also change the account credentials required for each country. For example, IBAN for Belgium, or BBAN and BSB code for Australia.

  • Delivery

    • Frequency: Required, how often the payment is made. Once for a singular payment.

    • Execution date: Required, The date the payment shall be made.

    • Standard and Urgent delivery selection: Required, displayed with cost and time taken to deliver payments.

The bank customer may click the X Clear button next to the Transfer to section of the payment form to remove entered beneficiary details without clearing the entered information in From account and Payment details.

For the international transfer type, the user may enter a Purpose of payment. The Purpose of payment field will have different formats depending on the country of the payment beneficiary, these are:

  1. An optional free text field.

  2. A mandatory free text field.

  3. A mandatory field, selected from a drop-down list of codes.

    1. The user is sees an explanatory description of each of the available codes.

The format is determined by receiving the sanctioned countries endpoint.

For certain countries, a Purpose of payment is a mandatory requirement, and some will require specific responses from the pre-defined list in the drop-down. In others, the Purpose of payment is entirely optional and can be authored by the payee.

  1. An optional free text field.

  2. A mandatory free text field.

  3. A mandatory field, selected from a drop-down list of codes.

Once the account user has entered all required details, the available payment type, or types if there are multiple, display. In the case that there are multiple payment types and the required fields for the payment types are different, such as ACH Credit or a domestic wire, the user is asked to select from different delivery options for each displayed payment type. Once the user selects a delivery option, the user is shown additional fields specific to that payment type. The user will see a cost overview for the transfer and the cost of selected delivery option. If available, the user is also shown a fee sharing option.

The account user has the option to add their own description of their payment for reference purposes.

Features

  • Single universal user interface guides the user to initiate a payment.

  • Supports SEPA CT, Faster Payments, CHAPS, International transfer types.

  • Select from saved contacts or enter new payment details.

  • Save payment details as a new contact.

  • Set up a standing order.

  • Save a created payment as a payment template for future payments.

  • Search for payment template.

  • Select and apply saved payment template to complete a new payment.

Initiate omni payment

The Initiate omni payment widget is a base widget that is fully customizable with a TypeScript configuration to support different payment types. It comes with out-of-the-box configurations for common payment types.

Supported payment types

The widget supports the following payment types out of the box:

  • Internal transfer

    A single payment type for transferring funds between the accounts of the same bank customer in the same financial institution.

    Use this payment type if you do not require specific validations for different internal account types, for example, checking, savings, or else. You can implement your own logic to filtering of the account types in the To and From drop-downs, as well as prohibiting transfers between certain account types. See widget reference, Internal account selector component, for configuration.

  • Intrabank transfer

    A payment type to allow a bank customer to transfer money to someone in the same bank.

    When initiating an intrabank transfer, the bank customer can save the manually entered name and account number as a new contact by selecting the Save to contacts checkbox. Once the payment is executed, the bank customer sees a notification in the success screen, confirming whether the contact was successfully saved.

  • A2A external transfer

    Payment type for transfers between accounts that the customer has in the different financial institutions. The available transfers allow users to:

    • Transfer money from an internal account to a connected account

    • Pull funds from a connected account to an internal account

    See widget reference, Internal account selector component, for configuration.

Features

When making a payment, the bank customer specifies the To and From accounts and the amount, adds an optional memo, and sets the payment schedule. All those fields are configurable. See widget reference for details.

By default, the widget displays currency symbols (for example, $) for To and From account balances. However, developers can configure the widget to display currency codes (for example, USD) instead.

When a bank customer selects a loan or a credit card account in the To drop-down, the widget displays a Payment details section with predefined payment amounts mapped to backend attributes. For example, a minimum due payment for a credit card account. The customer can select from the predefined options or enter an amount manually. See Initiate omni payment for the list of the OOTB mapped backend attributes.

The Schedule section allows the bank customer to select the date and frequency of the transfer.

For convenience, the OOTB configuration of the Schedule section is the following:

  • For credit card accounts, the widget displays a switch to pay monthly for any predefined payment amount. If the bank customer switches it on, then the payment frequency is automatically set to Monthly. Otherwise, it is set to Once.

  • For loan accounts, the widget displays the switch to pay monthly for the Other amount option (manual entering). If the bank customer switches it on, then the payment frequency is automatically set to Monthly. For full and principal amount options, the widget displays only the execution date and automatically sets the frequency to Once.

  • For other types of accounts, the widget displays the regular Schedule component with the date picker and frequency drop-down containing the Once, Daily, Weekly, Biweekly, Monthly, Quarterly, and Annually options.

When the customer selects a credit card or a loan account in the To drop-down, the widget compares the current date and the payment due date, and displays corresponding messages. The message can be either a blue info box, if the due date has not passed yet, or a yellow warning, if the payment is already late. The title, text, and icon of the messages are configurable.

Once the customer selects a payment date, the widget compares this date to the due date and shows a text warning about a late payment fee under the Execution date field.

Once all the information is provided, the bank customer reviews the payment details and confirms it.

If the payment type is an internal transfer or an intrabank transfer, the bank customer can edit previously scheduled payments. The bank customer initiates this the process in the Manage upcoming payments widget, and makes the changes in the pre-filled payment form of the Initiate omni payment widget. All fields are editable. Once the bank customer submits their changes, they see a success screen with an option to navigate back to the list of upcoming payments in Manage upcoming payments.

For a smooth experience, a developer can set up navigation from an accounts widget (for example, View retail account list), where the customer selects an account for transfer, and the Initiate omni payment widget. Initiate omni payment listens to the transferFrom parameter that contains the selected account ID, and preselects this account in the From drop-down.

Developers customize the widget with either out-of-the-box, or custom configurations. See: Adopt the Initiate omni payment widget.

Endpoints

This widget uses the following endpoints out of the box:

  • Retrieve internal accounts

    GET https://ips_host:ips_port/gateway/api/arrangement-manager/client-api/v2/productsummary/context/arrangements

  • Retrieve connected accounts

    GET https://ips_host:ips_port/gateway/api/payment-order-a2a/client-api/v1/external-accounts

  • Execute payment

    POST https://ips_host:ips_port/gateway/api/payment-order-service/client-api/v2/payment-orders

  • Update a payment order

    PUT https://ips_host:ips_port/gateway/api/payment-order-service/client-api/v2/payment-orders/{paymentOrderId}

Quick transfer

In the Quick transfer widget, the bank customer transfers money by providing minimum required information, which includes the following fields:

  • From: Sending US account.

  • To: Receiving US account.

  • Amount: The amount of transfer.

For the From account, the customer can select one of the following:

  • An internal account

  • A connected account

For the To account, the customer can select one of the following:

  • An internal account

  • A connected account

  • A contact account, that is, an account of another customer of the same bank

In the out-of-the-box configuration, transfers between connected accounts are turned off.

A developer can configure the following features of the widget:

  • Accounts shown in the From and To account drop-downs.

  • Allowed From and To account type combinations.

  • Required business functions.

  • Currency in the amount input field, before the From account is selected.

  • Payment types. See reference documentation for available OOTB payment types.

Features

This widget provides the following features for the bank customer:

  • Select a sending account from the customer’s internal accounts or connected accounts.

  • Select a receiving account from the internal, connected, or contact accounts.

  • Review payment details.

  • Confirm or discard the payment.

  • See a success or error message depending on the success or failure of the operation.

Connect external account

In this widget, the bank customer connects an external account that they hold in a different bank to make an external A2A transfer in the United States.

The bank customer specifies the following information to connect an external account:

  • Account number

  • Account type

  • Routing transit number (ABA)

  • Account name or alias

Once the external account is connected, the bank customer can view it in the list of accounts with the status Pending Activation. To be able to make an external A2A transfer, the bank customer must activate their previously connected external account. To do so, the bank customer enters two different micro-deposit amounts greater than 0 cents and less than or equal to 50 cents, and clicks Activate. The sum of these micro-deposits must not exceed 99 cents, and the two amounts must not be equal.

To cancel the activation process, the bank customer clicks Cancel, and confirms the action in the subsequent modal dialog. Upon successful activation, the customer sees a confirmation message, and is navigated back to the list view, where the updated status of the external account is visible. If the activation attempt failed, the customer is notified by an error message.

To remove a connected external account that is no longer required, the customer clicks Remove. It is possible to remove an account in any status, for example, if it is Pending Activation, or Failed Verification.

To view the details of an external account, the customer selects the account in the list. The following details are available:

  • Account type

  • Account number

  • Bank name

  • Routing transit number (ABA)

  • Account name or alias

  • Status

To transfer funds to and from a connected external account, the bank customer selects the make transfer option.

Features

  • Connect an external account

  • Activate a connected account

  • Remove a connected account

This widget interfaces with the Payments DBS capability using the Payments client API.

Request stop checks

In this widget, the bank customer submits requests to the bank to stop check payments.

If the bank customer issued a check, but wants to request the bank to not honor the payments, they submit a request to stop the clearance of an issued or lost check. This feature is helpful in the following scenarios:

  • The customer is short of funds, and wants to avoid the scenario of a bounced check.

  • The customer lost a signed check, and wants to prevent the bank from honoring it.

  • The customer lost an entire check book, and requests a bank to not honor checks that fall within a range of serial numbers.

To submit a stop check request, the bank customer fills in the check details form. If the request concerns a single check, the customer selects Single. If the request concerns multiple checks, the customer selects Multiple, and specifies the range of serial numbers.

To specify the reason of the request, the customer has the following options to choose from in a drop-down:

  • Insufficient funds

  • Incorrect amount or other details

  • Lost or stolen checks or check books

  • Checks mailed to wrong address

  • Dispute over purchases

  • Lost in mail

  • Other

Once the check details form is completed, the bank customer clicks Continue, and reviews the details of the request. To confirm the request, they click Submit.

Features

This widget has the following features:

  • Submit a stop check request for one or multiple checks.

  • Review the details of the request.

Manage upcoming payments

The Manage upcoming payments widget displays all the future dated one-off payments and standing orders in a list for a selected account.

The list is sorted in a descending order by requested execution date.

By default, the following information is available in the list:

  • Execution Date: The requested debit date of the payment.

  • Counter Party Name: The name of the recipient of the payment order.

  • Recurring Frequency: Frequency of recurrence of the payment order, if a payment order is submitted as recurring.

  • Instructed Amount: Amount of the payment order.

To view more information about a payment order, the bank customer clicks the payment order in the list.

Additional information can include:

  • Paying From Account: The account from the which the payment will be debited.

  • Counter Party Name: The name of the recipient of the payment order.

  • Counter Party Account: The account number of the recipient of the payment order.

  • Execution Date: The requested debit date of the payment.

  • Instructed Currency: Currency of the payment order.

  • Instructed Amount: Amount of the payment order.

  • Reference: Payment reference of the payment order.

  • Start date: The first requested execution date, if a payment order is submitted as recurring.

  • End date: The end date of a standing order, if set.

  • Recurring Frequency: Frequency of recurrence of the payment order, if a payment order is submitted as recurring.

  • Description: Free text description, if provided in the payment during submission.

To cancel a payment order or a payment schedule, the bank customer clicks the Cancel Payment button.

To edit an intrabank or an internal payment, the bank customer selects the Edit button. The bank customer is then redirected to the pre-filled payment form of the Initiate omni payment widget. All fields are editable. Once the bank customer submits their changes, they see a success screen with an option to navigate back to the list of upcoming payments.

Features

  • View a list of future dated payment orders for a selected account.

  • Load payment orders incrementally.

  • Display complete details of a payment order if selected from the list.

  • Display the currency exchange rate in payment details if the currency of the instructed amount is different from the currency of the debtor account.

  • Cancel a one-off upcoming payment before execution.

  • Cancel the payment schedule when canceling a standing order.

  • Edit scheduled internal and intrabank transfers.

This widget interfaces with the Payments DBS capability using the Payments client API.
