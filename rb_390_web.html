This release provides the following new features:

  • Web: Pagination property added in Experience Manager for View and download retail statements widget

    The bank customer is now able to update the pagination type for View and download retail statements widget through Experience Manager. The following options are supported:

    • Load more - set as default option

    • Pagination

Bug fixes

This release includes the following bug fixes:

  • MAINT-11162: Retail Account Statements - selecting date gives a wrong error message to the user

    The Invalid date error message, which was displayed when the bank customer selected only one date from the date picker in View and download retail statements widget, has been removed as it was incorrectly displayed.

This release includes the following deprecations:

  • In v3.9.0:

    • bb-account-statement-table-head component used in View and download retail account statements widget was removed. This release note lists the changes to all DBS capabilities in this release.

New features

This release provides the following new features:

  • Support for OpenAPI

    Widgets from all capabilities in this collection now support OpenAPI specifications. This change applies to newly generated web clients.

Deprecations

This release includes the following deprecations:

  • In the version released in January 2022:

    • Support ceases for Microsoft Internet Explorer 11

      Note
      		 This date was previously set to January 2021, but support for Microsoft Internet Explorer 11 will now continue to January 2022.

This release includes the following breaking changes:

  • All web widgets have been upgraded to version 3.9.0 of the data-ang project, which supports OpenAPI.

Authorized users v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: New widgets to create and manage authorized users

    The Create authorized user and View authorized user widgets have been created. These widgets enable bank customers to provide other individuals with limited or full access to their accounts. An authorized user (also known as sub-user) is the result of an account owner creating a set of credentials and respective permissions to enable a third-party to access their accounts. Possible use cases include: a parent giving their child limited access to accounts to view balances, or a retail customer providing an accountant with read-only access to their accounts.

Contacts v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Additionally manage contacts with email and telephone number

    In order to support P2P payments, the bank customer can now create and manage contacts including email and phone number as an account identifier. Banks can configure the options they support in Experience Manager. The options are BBAN, IBAN. email, and phone number.

Notifications v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: A human-readable timestamp

    A human-readable timestamp was added as an option to the Notification badge widget. This feature can help identify the needed notification among multiple notifications throughout the day.

  • Web: Show more

    The Show more feature was added as an option to the Notification badge and View sticky and popup notifications widgets. This change allows the bank customer to read the full text of the notification if the notification navigates to an external widget. It also solves the issue with the customer not being able to read the long text of a notification in a pop-up when the pop-up timeout is exceeded.

  • Web:Accessibility improvements

    The date of the notification is now readable for screen readers.

  • Web: Navigation pattern change

    Previously, the notification about a low retail account balance and a new transaction both had navigation provided into the active tab of the account details page. This was changed as follows:

    • The navigation from the low retail account balance is now configured to lead to the Account info tab of the account details page.

    • The navigation from the new transaction notification is now configured to lead to the Activity tab of the account details page.

    See Configure navigation for Notifications widgets.

Bug fixes

This release includes the following bug fixes:

  • MAINT-10241: Retail Banking Web Apps 2020.08-beta - Notifications take focus causes disruption for users

    The solution was added which allows reading of the notification without changing of the focus. This feature helps visually impaired app users to avoid distractions from their current tasks by new pop-up notification, and at the same time to be informed about new notifications.

Deprecations

This release includes the following deprecations:

  • In v3.10.0:

    • Support ceases for the isTextTruncated method of the bbNotificationsListItemCustomizable slot in the Notification badge widget.

Breaking changes

This release includes the following breaking changes:

  • The old navigation approach for widgets is replaced with the new navigation approach. See Configure communication between widgets.

Payments v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Pre-selected account in Initiate omni payment

    The Initiate omni payment widget now accepts the transferFrom parameter. This allows to have the From account pre-selected when the bank customer enters Initiate omni payment after selecting an account in a different widget.

  • Web: New features in Quick transfer

    • The widget now supports the following payment types:

      • INTERNAL_TRANSFER for transfers between the customer’s own accounts in the same bank.

      • EXTERNAL_A2A for transfers to and pulling funds from connected accounts in a different bank.

        See reference documentation.

    • In addition to transferring to connected accounts, bank customers can now pull funds from connected accounts to their internal accounts. See functional documentation.

    • The payment type and the filtering logic for the To and From account selectors are now configurable. See reference documentation.

    • Developers can now show and hide the Contacts tab in the widget. If on, the tab displays the accounts of contacts as payment recipients. For this to work, implement an additional payment type to handle payments to contacts. This payment type is not available out of the box. In the OOTB widget configuration, the tab is hidden.

  • Web: Removed edit functionality from Manage upcoming payments

    The Edit button has been removed from the upcoming payment details screen because this functionality is not supported by DBS.

  • Web: Transaction signing in create payment flow

    In the omni payment and payment wizard widgets, the user confirms a submitted payment by a chosen transaction signing method.

    • User is asked to confirm via a new dialogue with the confirmation challenge and other payment details.

    • User can enter this confirmation on the same device, in-band, or another device, out-of-band, as per user’s choice.

    • User can see a message for successful confirmation as well as sub-sequent successful submission of payment.

Bug fixes

This release includes the following bug fixes:

  • MAINT-10309: Retail Banking Web App Beta: bb-payord-omni-payment-widget-ang required fields are not indicated

    Required fields are now marked as required or aria-required.

  • MAINT-10310: Retail Bank Web App Beta: bb-payord-omni-payment-widget-ang amount input has bad aria-label, not associated with visible label

    • The aria-label of Number Input is now descriptive.

    • The visible label of Amount input is now associated correctly with the label in the DOM.

    • The sr-only label is removed.

  • MAINT-10313: Retail Bank Web App Beta: bb-payord-omni-payment-widget-ang aria-describedby is not working

    The aria-describedby property is now wired up correctly.

  • MAINT-10304: Retail Banking Web App Beta: bb-billpay-manage-payee-widget aria-describedby form id not presented for validation errors

    Now validation errors are read as part of the description when the focus is on the field.

  • MAINT-10305: Retail Banking Web App Beta: bb-billpay-manage-payee-widget required fields are not indicated

    Required fields are marked as required or aria-required.

  • MAINT-10307: Retail Banking Web Apps 2020.08-beta - bb-billpay-payments-list items are not focusable

    Date, name of the payee, status, inline text, and amount are now working correctly in terms of accessibility.

Deprecations

This release includes the following deprecations:

  • In v2.21.0:

    • Support ceases for the following payment types:

      • CURRENT_TO_SAVINGS

      • SAVINGS_TO_CURRENT

      • CASH_ADVANCE_CURRENT

      • CASH_ADVANCE_SAVINGS

      • LOAN_ADVANCE_CURRENT

      • LOAN_ADVANCE_SAVINGS

      • CHECKING_TO_CHECKING

      • SAVINGS_TO_SAVINGS

    Use the single INTERNAL_TRANSFER payment type for all kinds of internal transfers.

Breaking changes

This release includes the following breaking changes:

  • Removal of 'Initiate Payment' widget. Initiate payment (Deprecated in 3.8.0) is removed from Retail Banking Collection.

Products v3.9.0

This release note explains changes to this capability.

Bug fixes

This release includes the following bug fixes:

  • PRSUM-6208: Wrong font-weight and font-size for bb-amount

    When clicking on the account picker, the bb-amount had font-weight 400 whilst it should have font-weight 600 and the bb-amount font-size of 16px whilst it should be font-size of 18px. This has now been fixed.

  • MAINT-10322: bb-product-summary-manage-accounts-widget edit nickname aria-label insufficient

    The Edit label is now made clearer as Edit alias.

  • MAINT-10321: bb-product-summary-details-widget excess tab stops

    Each card in the Product summary details widget had a tabstop, and each heading within some of the cards also had a tabstop. These aren’t needed as the elements are not intended to be interacted with, and have been removed.

  • MAINT-10248: bb-product-summary-list-widget inconsistent bolding of owner names

    Owner names sometimes appeared in bold, and in other cases in normal weight. All instances of bb-product-item_owners now use font-weight regular, which is configured as weight 400.

  • MAINT-10247: bb-product-summary-list-widget product kind item button screen reader experience

    Entire content of the product card is read, along with the button when using a screen reader, and the user may have a difficult time understanding the specific information within the card, as the elements contained within the button are not represented as discrete elements which can be navigated through.

  • MAINT-10246: bb-product-summary-list-widget favorite status does not have a text alternative

    Text alternative for the blue start favorite status icon was not available. The icon now has a tooltip, and can be accessed with tabbing to view. The Favorite Account is also read by screen reader when tabbing.

  • MAINT-10245: bb-product-summary-list-widget has incorrect tab role

    The bb-product-summary-list-widget had role="tab" on each product kind, although it wasn’t a tab. Screen readers misidentified the element and anounced its contents as its label. The tab role is no longer read by screen reader. The button is read instead, as suggested by W3.org.

  • MAINT-10678: Double submission of change password form on submit

Migration

When you migrate from one version of this product to another, best practice is to update incrementally through all minor and major versions in between.

Follow these steps to update an existing deployment:

  1. Update non-prefixed references in the data-ang project

    To move from the deprecated non-prefixed ApiModule, BASE_PATH, and ConfigurationParameters, follow these steps:

    1. Check the public_api.ts file in data-ang project for the correct prefix to use. For example, the Manage user security widget requires the prefix User.

    2. Update your project to use the prefix. For example, replace ApiModule, BASE_PATH, and ConfigurationParameters with UserAPIModule, USER_BASE_PATH, and UserConfigurationParameters.
