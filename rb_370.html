All capabilities v3.7.0

This release note lists the changes to all capabilities in this release.

Announcements

This release includes the following announcements:

  • Renamed widget collection

    Widget Collection Retail Banking is now named Retail Banking Widget Collection. This name change is only for the documentation, and does not have any impact on the content of the product.

Deprecations

This release includes the following deprecations:

  • In 3.7.1:

    • Support ceases for the triangular services for a number of backend capabilities. The widgets in this collection are updated to use the new lean services.

      The following Digital Banking Services capabilities have implemented the lean architecture this release:

      • Access Control

      • Account Statements

      • Audit

      • Bill Pay

      • Personal Finance Management

      • Products

      • Transactions

      • Users

      The following Entitlements Services capabilities have implemented the lean architecture this release:

      • Access Control

      • Limits

      • Users

      To see how to roll back the lean architecture and use the deprecated triangular architecture, see Change service type.

  • In 3.8.0:

    • Support ceases for Angular 8. All Angular widgets, journeys, and libraries will upgrade to Angular 10.

  • In the version released in January 2021:

    • Support ceases for Microsoft Internet Explorer 11

      Note
      		 The date for dropping support for Microsoft Internet Explorer 11 has changed. Support will now continue to January 2022.

Breaking changes

This release includes the following breaking changes:

  • End of support for iOS 11 and Android 4.4.

Contacts v3.7.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Retail Contact Manager to support Account Number (BBAN)

    Now it is also possible to configure an Account number (AKA BBAN - Basic Bank Account Number) for a contact. A contact can be entered with IBAN or Account number based upon configurations in the widget. These configurations can be achieved using CXP manager. There are no validations performed for an entered Account number. An Account number can be of maximum 34 characters.

Notifications v3.7.0

    • Manage and count notifications widget was renamed to Notification badge widget.

    • Manage business approval notification settings widget was renamed to Manage business approval notification preferences.

  • Navigation to target widget

    It is now possible to set up clickable notifications that navigate the reader to the relevant widget in the following manner:

    • From the new message notification to the relevant message thread

    • From the change balance and new transaction notification to the relevant account details

    See Configure navigation for Notifications widgets for setup.

    The notification recipient can now mark all notifications as read in the mobile Manage notifications widget.

Payments v3.7.0

This release note explains changes to this capability.

New features

This release includes the following new features:

  • Internal transfers

    • Transfer money between own accounts within the same financial institution by using a configuration for the Initiate omni-payment widget.

    • The CHECKING_TO_SAVINGS payment type is implemented in the form.

    • View the amount input field with the currency, amount, and decimals in one.

    • Configure the available/current balance labels in the product selector.

See Initiate omni payment.

  • Quick transfers

A new widget that allows to transfer funds from an internal account to another internal, external, or a contact’s US account with minimum fields. To complete the quick transfer, the bank customer selects the sending and receiving accounts, enters the amount, and then reviews and confirms the transfer.

See Quick transfer.

Bug fixes

This release includes the following bug fixes:

  • MAINT-9631: Payment Widget Extension slots code is incorrect.

    The template for the Initiate payment bbPayordInitiatePaymentFormDebitorCustomizable extension slot was incorrect and threw the openPopup is not a function error. The template has been updated with the latest working code in the documentation.

Breaking changes

This release includes the following breaking changes:

  • ProductSummaryClient is now removed in favor of ProductsClient.

    For this reason, the following breaking changes apply for both Android and iOS:

    • The product property in InitiatePaymentOrderWrapper class is removed in favor of the selectedProduct property.

    • The updateViewWithProduct method in PaymentOrderView is removed in favor of the updateViewWithSelectedProduct method.

  • The ui-ang split out of foundation-ang.

    Starting from foundation-ang@5.0.0, @backbase/ui-ang will no longer be a part of the collection. It would be required to add @backbase/ui-ang as a separate dependency for projects and change imports containing the path @backbase/foundation-ang/ui to @backbase/ui-ang. Check foundation-ang release notes for updating using schematics and more details.

To update an existing deployment to accommodate these breaking changes, see the Migration guide.

Transactions v3.7.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Ability to view the merchant name and icon of a transaction

    The View transactions list widget now provides the ability to display the merchant icon and name of a transaction so that bank customers can easily identify their transactions.

  • Web: Ability to have a more comprehensive visual indication of transactions with a new set of icons

    The View transactions list widget now provides the ability to display different icons for transactions based on available data. In addition to category icons, now transactions can display merchant icons, transaction type icons, incoming transaction icon and outgoing transaction icon. Which icon to be displayed depends on the following priority order: merchant icons > category icons > transaction type icons > Incoming and outgoing transaction icons. For example, if the merchant information is not present for a transaction, then category information is checked and an icon for category is shown, if it exists. If category information is also not present different icons are shown based on the transaction type. If transaction type information is not present, then either an incoming or outgoing transaction icon is shown.

  • Web: A refreshed transaction details view with a cleaner user interface

    View transaction details* now has a new look with a header which includes vital transaction information followed by more readable fields.

  • Web: Currency exchange details

    Currency exchange details are displayed in View transaction details in a more compact way with the introduction of the ‘currency exchange field.

  • Web: Running balance in View transactions list

    Running balance amount is now displayed in transactions list below the transaction amount.

  • Mobile: Download and share check images in transaction details.

    Customers now can easily share and download their check images in PNG or JPEG format, depending on the format of the image.

  • Mobile: View an enlarged view of check images.

    Customers can now enlarge their check images and swipe through them.

Breaking changes

This release includes the following breaking changes:

  • Mobile: ProductSummaryClient is now removed in favor of ProductsClient. For this reason, the following breaking changes apply:

    • Android:

      • The KEY_PRODUCT variable in TransactionsWidget is removed in favor of the KEY_SELECTED_PRODUCT variable.

      • The getProduct() method in TransactionsContract is removed in favor of getSelectedProduct().

      • The updateViewWithProduct(Product prouct) method in TransactionsView is removed in favor of updateViewWithProduct(BaseProduct selectedProduct) method.

      • In the class TransactionsDataImpl, public static final String TRANSACTION_PRESENTATION_SERVICE = "transaction-presentation-service"; changed to public static final String TRANSACTION_SERVICE = "transaction-manager";.

    • iOS:

      • The product property in TransactionsContract is removed in favor of selectedProduct property.

      • The loadTransactions method in TransactionsView is removed in favor of updateViewWithProduct method.

  • Web:

    • The defaultExportConfig input in BaseTransactionsControls is removed in favor of exportConfig input.

    • The transactionsList input in TransactionsNumberOfResultsFoundComponent is removed in favor of numberOfResultsFound input. The component now receives only the amount of results found.

    • Notification types NotificationMessage and NotificationModifier in TransactionsCategoriesComponent are no longer used.

    • ViewChild in TransactionsCategoriesComponent (notificationUpdate, notificationErrorUpdate) is no longer used.

    • Method pushNotification() in TransactionsCategoriesComponent is no longer used.

    • Output cancelCategory in TransactionsCategoriesComponent replaced with categoryChange.

    • Method cancelChangeCategory() in TransactionsCategoriesComponent is replaced with cancelSaveCategory().

    • Method saveCategory() in TransactionsCategoriesComponent is replaced with changeCategory().

    • The getter unsignedAmount in TransactionsTableDetailsComponent is removed. The transaction object now has a property called transactionAmountCurrency that in turn has unsignedAmount value.

    • The transaction model went through the following changes:

      • Removed productId property.

      • Removed amount property in favor of transactionAmountCurrency.amount.

      • Removed currency property in favor of transactionAmountCurrency.currencyCode.

      • Removed scheduledDate property.

      • Removed instructedAmount property in favor of instructedAmountCurrency.amount.

      • Removed instructedCurrency property in favor of instructedAmountCurrency.currencyCode.

      • Removed accountNumber property.

      • Removed debitCreditSign property.

To update an existing deployment to accommodate these breaking changes, see the Migration guide.

Migration

When you migrate from one version of this product to another, best practice is to update incrementally through all minor and major versions in between.

Follow these steps to update an existing deployment:

  1. Changes for new lean services in native mobile clients

    In DBS version 2.19.0, the Transactions and PFM service names changed.

    • If your DBS instance is version < 2.19.0 and you decide to use TransactionsClient 3.7.0, then make sure that your implementation still uses transaction-presentation-service as the service name. In version 3.7.0 of TransactionsClient, the default service name used by TransactionsDataImpl becomes transaction-manager.

      • Android

        val dbsBaseUrl = "127.0.0.1" //replace this ip with your dbs ip or url
val serviceName = "transaction-presentation-service"
val dbsClientPath = DbsClientPath(dbsBaseUrl, serviceName)
val transactionParser: TransactionsParser = DefaultTransactionsParser()
val dataImpl: TransactionsData = TransactionsDataImpl(transactionParser, Backbase.getInstance(), dbsClientPath)
val dataProvider: DBSDataProvider = NetworkDBSDataProvider(context)
val transactionClient = TransactionsClient(dataProvider, URI(dbsBaseUrl), dataImpl)

      • iOS

        let serverUrl = URL(string: "127.0.0.1") //replace this ip with your dbs ip or url
let clientPath = BBDbsClientPath(serviceName: "transaction-presentation-service", withBaseURIVersion: "v2")
let transactionsClient = TransactionsClient(dbsClientPath: clientPath)
transactionsClient.baseURL = serverURL

    • If you upgrade DBS to version 2.19.0, then upgrade the TransactionsClient to 3.7.0. Make sure your implementation uses the new transaction-manager service name.

      • Android

        If you, even indirectly, use TransactionsDataImpl, then you do not need to manually set the service name.

    val dbsBaseUrl = "127.0.0.1" //replace this ip with your dbs ip or url
val dataProvider: DBSDataProvider = NetworkDBSDataProvider(context)
val transactionClient = TransactionsClient(dataProvider, URI(dbsBaseUrl)) //by using this constructor, an instance of TransactionsDataImpl is created

    • iOS

      The default initializer for TransactionsClient is using the new transaction-manager service name.

      let serverUrl = URL(string: "127.0.0.1") //replace this ip with your dbs ip or url
let transactionsClient = TransactionsClient()
transactionsClient.baseURL = serverURL

      • If your DBS instance has version < 2.19.0, and you decide to use PfmClient 3.7.0, then make sure your implementation still uses the old service names. Otherwise, your requests will use URLs that are not mapped by your DBS instance. In version 3.7.0 of PfmClient, the default service names used are transaction-category-collector, budget-planner, and saving-goal-planner.

    • Android

      val dbsBaseUrl = "127.0.0.1" //replace this ip with your dbs ip or url

val categoriesServiceName = "categories-management-presentation-service"
val dbsClientPathForCategories = DbsClientPath(dbsBaseUrl, categoriesServiceName)

val budgetsServiceName = "budgeting-presentation-service"
val dbsClientPathForBudgets = DbsClientPath(dbsBaseUrl, budgetsServiceName)

val savingGoalsServiceName = "saving-goals-presentation-service"
val dbsClientPathForSavingGoals = DbsClientPath(dbsBaseUrl, savingGoalsServiceName)

val pfmParser: PFMParser = DefaultPFMParser()
val dataImpl: PFMData = PFMDataImpl(Backbase.getInstance(), pfmParser, dbsClientPathForCategories, dbsClientPathForBudgets, dbsClientPathForSavingGoals)
val dataProvider: DBSDataProvider = NetworkDBSDataProvider(context)
