Understand the search bar framework

Some widgets (e.g. View transactions, turnovers, and spending/income analysis ) feature a search capability. The use case of the capability depends on the widget. For example, the Transactions widget uses it to search for specific transactions. While the use case can be different, the way a user uses the search capability typically is not. This ensures a consistent way of working across the entire mobile app. This page describes a pattern used by mobile widgets developed by Backbase to support searching.

The search capability of mobile widgets relies on a generic implementation of a search bar and a specific implementation per widget. These implementations communicate with each other using pub/sub events. The following illustration provides an example of both implementations, which is described in more detail below.

  1. Showing the Search Bar

    The mobile app (hosting the widgets) should have a generic search bar that is shown only when the rendered page has a preference with name search. The value of this preference defines for what widget the search bar is enabled. This is described in the next section.

  2. Configuring the Search Bar

    As the search bar is generic, it could potentially affect multiple widgets in a mobile app containing a search capability. In order to specify for what widget the search bar should be enabled in order to avoid unexpected behaviour, the search preference described previously should be set to match a predetermined value for a specific widget. For Backbase mobile widgets this predetermined value can be found in the documentation for each widget specifically.

  3. Search Bar Interaction

    The search bar interacts with the search capability of a widget using pub/sub events. The following interactions are supported:

    • The search bar receives focus - When a user activates the search bar triggering it to receive focus, an event is sent without a payload with a name that is formatted like this, where widget refers to the value of the search preference defined earlier:

      bb.action.<widget>.search.open

      For example:

      bb.action.places.search.open

      This event can be used by a widget to prepare itself for a user searching for something. For example to show an auto-completion list.

    • The search bar loses focus - When a user deactivates the search bar causing it to lose focus, an event is sent without a payload with a name that is formatted like this, where widget refers to the value of the search preference defined earlier:

      bb.action.<widget>.search.close

      For example:

      bb.action.places.search.close

      This event can be used by a widget to close its search capability. For example by hiding an auto-completion list.

    • The value of the search bar changes - When a user changes the value of the search bar, an event is sent with a payload containing the updated value of the search bar. The event name is formatted like this, where widget refers to the value of the search preference defined earlier:

      bb.action.<widget>.search.changed

      For example:

      bb.action.places.search.changed

      The payload of this event is a map (Android) or dictionary (iOS) with key value set to the updated value of the search bar as a string.

      This event can be considered the main event of the search capability as it should trigger the search capability to perform an actual search action.

Widget Interaction

While most interaction is performed from the search bar, in some cases a widget requires updating the value of the search bar itself too. In order to support this, the search bar listens for an event named like this, where widget refers to the value of the search preference defined earlier:

bb.action.<widget>.search.set

For example:

bb.action.places.search.set

The search bar expects a payload as a map (Android) or dictionary (iOS) with key value set to the requested new value of the search bar as a string. The search bar will update the text shown without triggering the bb.action.<widget>.search.changed event.

This event is typically used to confirm a selection after a user selected a search result. For example, the Places widget might show search result "Amsterdam" when the user searches for "Ams". Selecting this search result updates the value of the search bar to "Amsterdam".

FAQ

I want to use a search bar within a widget, not outside of the widget

While this page describes the concept of a generic search bar that can be configured to work with specific widgets, an implementor might require the search bar to be (visually) within the widget itself. This can be done easily by customizing the user interface of the widget to contain a search bar triggering similar events as described above. The page hosting the widget should not have the search preference as this would cause the search bar to be shown on activity/view controller level too.
