Submit remote deposit check

Note
 This section describes an add-on. You may use this add-on only if you have purchased an appropriate license for this add-on based on a signed Product Order Form.

Views

This widget is developed following the Native Renderer Blueprint. As a result, this widget consists of a view that can be replaced as long as it meets the view contract defined by the widget. These APIs might use the delegation pattern for asynchronous tasks such as loading external data.

Refer to the API reference of this widget for more information about the contracts, available APIs and their usage.

The class name of this widget is RDCFormWidget.

In order to render an instance of this widget to a page, apply the following preferences to that page:

  • native: RDCFormWidget

  • native.view: NameOfYourClassWidgetView (implementing RDCFormView)

  • native.package: package.containing.your.view (only needed for Android)

For an example of the widget usage, check the sample code from repo.backbase.com

Preferences

This section provides an overview of the preferences that control the behavior of the widget.

  • mode: This preference is used by the widget to determine which pub/sub events should be listened to. This preference is for letting the widget know which pub/sub events it should listen to. They are used by the widget in order to trigger certain functionality. If no mode is specified, the widget defaults to the initiate mode.

    • initiate: This mode is used when capturing a new remote deposit.

    • review: This mode is used for reviewing the validated remote deposit before submission. This mode works in association with DefaultRDCFormReviewView (or the class name of your custom view).

    • submitFailed: This mode is used when submitting the deposit fails. This mode works in association with DefaultRDCSubmissionFailedView (or the class name of your custom view).

    • submitSucceeded: This mode is used when the deposit has been successfully submitted. This mode works in association with DefaultRDCSubmissionSucceededView (or the class name of your custom view).

Events

This section provides an overview of the pub/sub events used by the widget.

Published events

  • bb.action.rdc-form.product-requested: Published when the bank customer taps the product selector, indicating they want to select a product. It is compatible with the SelectorWidget in order to easily support a user flow. The JSON payload contains the RDCProduct object:

  • bb.action.rdc-form.images-requested: Typically published to trigger the camera view to allow the bank customer to scan a deposit. The resulting image is expected to be returned to this widget via the bb.event.rdc-form.front-image-captured and bb.event.rdc-form.back-image-captured events described below.

  • bb.action.rdc-form.front-image-requested: Published when the bank customer taps on the front image, indicating they want to request a new front image of the check.

  • bb.action.rdc-form.back-image-requested: Published when the bank customer taps on the back image, indicating they want to request a new back image of the check.

  • bb.action.rdc-form.deposit.entered: Published when the bank customer taps the Continue button, indicating they want to move to the next step after selecting a product, entering the deposit amount, and providing the images of the check. The payload contains a JSON serialization of the RDCFormData object.

    If the review mode is implemented in the model, the navigation preference must match this event name:

    {
    "name": "navigation",
    "value": "bb.action.rdc-form.deposit.entered"
}

  • bb.event.rdc-form.deposit.submission.failed: Published when the form submission has failed with validation errors. For example, when the inserted amount doesn’t match the amount in the front image of the check. The payload contains an array of validation errors.

    If the submitFailed mode is implemented in the model, the navigation value must match this event name:

    {
    "name": "navigation",
    "value": "bb.event.rdc-form.deposit.submission.failed"
}

  • bb.event.rdc-form.deposit.submission.succeeded: Published when the form submission has succeeded. It must provide a JSON payload representing the Deposit object.

    If the submitSucceeded mode is implemented in the model, the navigation value must match this event name.

{
    "name": "navigation",
    "value": "bb.event.rdc-form.deposit.submission.succeeded"
}

Registered events

  • bb.action.rdc-history.new-deposit: Registered to launch the submission form in its default mode. It is compatible with RDCHistoryWidget in order to easily support the user flow.

  • bb.action.rdc-form.deposit.try-again: Registered to launch the submission form in its default mode. It sends a payload to pre-fill the form with the JSON representation of the RDCFormData object.

  • bb.action.product-selector.rdc.product-selected: Registered when the bank customer selects a product. It is compatible with the ProductSelectorWidget, which typically publishes this event, and must contain a JSON payload with the selected RDCProduct object.

  • bb.event.rdc-form.front-image-captured: Registered when the bank customer captures the front side of the check. The JSON payload contains the base64-encoded data bytes of a picture representing the front of a deposit check.

  • bb.event.rdc-form.back-image-captured: Registered when the bank customer captures the back side of the check. The JSON payload contains the base64-encoded data bytes of a picture representing the back of a deposit check.
