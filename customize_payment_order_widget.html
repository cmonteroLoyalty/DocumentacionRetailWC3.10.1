Customize the Create payment order widget

This section shows you how to:

  • Create a payment order for a specific payment type.

  • Validate the payment order for that specific payment type.

The supported payment types are:

  • SEPA Credit Transfer

  • US payments:

    • US domestic wire transfer

    • US foreign wire transfer (deprecated)

    • A2A external transfer to an from an external account

  • UK payments:

    • Faster Payments

    • BACS

The widget handles the view creation and the validation of the required fields based on these payment types.

See Understand Payments for more information about payment types.

Payment type selection

In order to select a supported payment type, execute the following implementation. For example, to create a SEPA payment:

Android

paymentOrderContract.selectPaymentType(SupportedPaymentType.SEPA_CREDIT_TRANSFER);

iOS

self.contract?.selectPaymentType(.SEPA)

You can skip using the selectPaymentType method and configure the payment type manually by publishing the bb.action.payment-order.select-payment-type pub/sub event:

Android

SEPAFormSpec sepaSpec = new SEPAFormSpec();
backbase.publishEvent(PaymentOrderWidget.EVENT_SELECT_PAYMENT_TYPE, backbase.getCurrentModel().getExperience().getId(), sepaSpec);

iOS

let sepaSpec = BBSEPAFormSpec()
let payload = ["paymentOrderFormSpec" : sepaSpec]
Backbase.publishEvent(PaymentOrderWidget.paymentOrderSelectPaymentTypeAction, object: self.contract as Any, payload: payload)

To learn how to set up the backend to support custom payment types, see Add a custom payment type. If you try to use the widget to create a payment of a custom type without configuring the backend first, you will get an error message. Once a custom payment type is added to the backend, the custom payment type can be selected in the widget as follows:

Android

PaymentSpec paymentSpec = new PaymentSpec("GBP", "GB_TRANSFER");
PaymentOrderFormSpec paymentOrderspec = new DefaultFormSpec(paymentSpec);
paymentOrderContract.selectPaymentType(paymentOrderspec);

iOS

let paymentSpec = BBPaymentSpec(currencyCode: "GBP", paymentType: "GB_TRANSFER")
let paymentOrderspec = BBPaymentOrderFormSpec(paymentSpec: paymentSpec)
self.contract?.selectPaymentType(with: paymentOrderspec)

Register view in the CXP model

If you select the payment type in a different view from where you create the payment order, then add following preferences to the view which will load your create payment view:

CXP model

{
  "mode": "initiate",
  "maxAmountDigits": "10",
  "navigation": "bb.action.payment-order.select-payment-type",
  "native": "PaymentOrderWidget",
  "native.view": "NameOfYourCustomView",
  "native.package": "package.of.your.custom.view" //This preference is mandatory for your android implementation.
}

If selection of the payment type is in the same view as the creation, then you can add the following preferences to your CXP model. These preferences include a custom navigation pub/sub event. The most important factor is that you are adding the mode as initiate so that all the pub/sub events for the initiate mode are registered in the widget.

CXP model

{
  "mode": "initiate",
  "maxAmountDigits": "10",
  "navigation": "flow.start.PaymentOrder",
  "native": "PaymentOrderWidget",
  "native.view": "NameOfYourCustomView",
  "native.package": "package.of.your.custom.view" //This preference is mandatory for your android implementation.
}

Load payment configuration in the create view

Once the view and the payment type are set up, render the view types for the selected payment type. When the bb.action.payment-order.select-payment-type pub/sub event is published (via selecting the payment type), the widget triggers the updateViewWithPaymentOrderFormSpec method. Therefore, you need to implement this method in the view.

Android

public void updateViewWithPaymentOrderFormSpec(@Nullable PaymentOrderFormSpec paymentOrderFormSpec) {
	if(formSpec != null) {
		setupViews(paymentOrderFormSpec);
	}
}

iOS

public func update(with paymentOrderFormSpec: BBPaymentOrderFormSpec?) {
	guard let paymentOrderFormSpec = paymentOrderFormSpec else {
        return
    }
	self.setupViews(with: paymentOrderFormSpec)
}

Optionally you can retrieve the view types that are required for this specific payment type. This view types are a list of PaymentOrderFieldType objects (for iOS BBPaymentOrderFieldType). With these view types you can create different types of list items.

Android

final void setupViews(@Nonnull paymentOrderFormSpec: PaymentOrderFormSpec) {
	PaymentOrderFieldType[] fieldTypes = paymentOrderFormSpec.getFieldTypes();
	System.out.println(Arrays.toString(fieldTypes));
}

iOS

func setupViews(with paymentOrderFormSpec: BBPaymentOrderFormSpec) {
    let fieldTypes = paymentOrderFormSpec.retrieveFieldTypes()
	print(fieldTypes)
}

Assign data

After creating the views assign the data via the user input. Since all properties of the InitiatePaymentOrder object are read-only, the widget provides an InitiatePaymentOrderWrapper class, which has all the fields as read-write properties.

Note that maxAmountDigits preference is set in the CXP model, and it is used to limit the payment amount digits. It is not validated by the widget, which means you need to validate this in your view logic.

Android

InitiatePaymentOrderWrapper initiatePaymentOrderWrapper =  new InitiatePaymentOrderWrapper(paymentOrderFormSpec, getContract().getMaxAmountDigits());

iOS

self.paymentOrderWrapper = BBInitiatePaymentOrderWrapper(paymentSpec: paymentOrderFormSpec, maxAmountDigits: self.contract?.maxAmountDigits ?? 5)

Validate

Once the views and data assigning are completed, validate whether the assigned data is correct in the following way:

  1. Create an InitiatePaymentOrder object out of InitiatePaymentOrderWrapper.

    The first part of the validation is done in this step, all the mandatory fields of the InitiatePaymentOrder object are checked.

  2. Validate each required data field according to the payment type.

    If validation fails, it will return an error object which has a validation status (for Android) or an error (for iOS), which includes the details of the validation failure.

  3. Review the validated payment order

    At this step, the payment order is also ready for submission. If you want to skip the review page, you can directly submit the payment order with the createPaymentOrder method.

Android

protected void validatePaymentOrder() {
	try {
		// Step 1
		InitiatePaymentOrder initiatePaymentOrder = initiatePaymentOrderWrapper.createInitiatePaymentOrder();

		// Step 2
        ValidationStatus validation = paymentOrderFormSpec.validate(wrapper.createInitiatePaymentOrder(), getContext());
    } catch (InvalidInitiatePaymentOrderException e) {
        BBLogger.error(TAG, e);
        validation = new PaymentOrderFormSpec.ValidationStatus(false, e.getMessage());
    }


	if (validation.isValid) {
		// Step 3
        getContract().reviewPaymentOrder(initiatePaymentOrderWrapper);
    } else {
        BBLogger.error(TAG, validation.message);
        showSnackbar(validation.message);
    }
}

iOS

func validatePaymentOrder() {
	do {
		// Step 1
	    let initiatePaymentOrder = try self.paymentOrderWrapper.createInitiatePaymentOrder()

		// Step 2
	    guard let error = self.paymentOrderFormSpec?.validate(self.paymentOrderWrapper) else {
			// Step 3
	        self.contract?.reviewPaymentOrder(with: initiatePaymentOrder)
	        return
	    }
	    self.presentError(error: error)
	} catch {
	    self.presentError(error: error)
	}
}

func presentError(error:Error) {
    let userInfo = (error as NSError).userInfo
    self.bb_parentViewController()?.presentSimpleAlert(withTitle: "Error", message: userInfo.description)
}

Review

In case your review screen is separate from the validation screen, then the review screen should be registered in the CXP model as follows:

CXP model

{
  "mode": "review",
  "navigation": "bb.action.payment-order.review",
  "native": "PaymentOrderWidget",
  "native.view": "NameOfYourCustomView",
  "native.package": "package.of.your.custom.view" //This preference is mandatory for your android implementation.
}

When you register bb.action.payment-order.review pub/sub for mode review, you need to implement the view update method, which is part of PaymentOrderView. This method will return the InitiatePaymentOrderWrapper object.

Android

@Override
public void updateViewWithPaymentOrder(@NonNull InitiatePaymentOrderWrapper initiatePaymentOrderWrapper) {
    this.initiatePaymentOrderWrapper = initiatePaymentOrderWrapper;
}

iOS

public func update(with initiatePaymentOrderWrapper: BBInitiatePaymentOrderWrapper) {
    self.initiatePaymentOrderWrapper = initiatePaymentOrderWrapper
}

This wrapper object can be used to recreate each view field and it also has a method called createInitiatePaymentOrder. This method can be used to recreate the InitiatePaymentOrder object, which is used in submitting the payment order in the submit step below.

Submit

To submit the data, you need the InitiatePaymentOrder object. As the view is updated with the wrapper object in the previous step, we can make use of the InitiatePaymentOrder object, which can be created by using the wrapper object.

  1. Create the InitiatePaymentOrder object by using the wrapper object and submit the payment order.

  2. Handle the received response.

    In successful response, the flow can end or proceed to the overview screen as follows:

Android

// Step 1
private void submitPaymentOrder() {
    InitiatePaymentOrder paymentOrder = initiatePaymentOrderWrapper.createInitiatePaymentOrder();
    getContract().createPaymentOrder(paymentOrder, new PaymentListenerImpl());
}

// Step 2
final class PaymentListenerImpl implements PaymentsListener {
    @Override
    public void onSuccess(PaymentOrdersPostResponse paymentOrdersPostResponse) {
        getContract().overviewPaymentOrder(initiatePaymentOrderWrapper, paymentOrdersPostResponse);
    }

    @Override
    public void onError(Response response) {
        showSnackbar(response.getErrorMessage());
    }
}

iOS

// Step 1
@IBAction func submitButtonPressed(_ sender: Any) {
  let paymentOrder = try? initiatePaymentOrderWrapper.createInitiatePaymentOrder()
  guard let paymentOrder = paymentOrder else {
    presentError(error: error)
    return
  }
  self.paymentOrder = paymentOrder
  self.contract?.createPaymentOrder(with: paymentOrder, delegate: self)
}

// Step 2
extension PaymentOrderWidgetReviewView: PaymentOrderDelegate {
    public func didCreatePaymentOrder(with paymentOrdersPostResponse: BBPaymentOrdersPostResponse) {
        self.contract?.overviewPaymentOrder(with: self.paymentOrder, paymentOrdersPostResponse: paymentOrdersPostResponse)
    }

    public func didFailCreatingPaymentOrderWithError(_ error: Error) {
        presentError(error: error)
    }
}

At this step the payment order creation flows end. The following section explains how to use other functionallity of the widget.

Select a sender account

To select a sender account for the payment order, you need to use the SelectorWidget. This section explains how to make use of the SelectorWidget to select the sender account.

Step 1: Declare the SelectorWidget in the preferences of the CXP model and add the client that will be used in the SelectorWidget. In this case, since the sender account needs to be selected, this account should be one of the accounts of the person who is using the application. That is why ProductSummaryClient is used. On the other hand, to show the Selector Widget bb.action.payment-order.select-product pub/sub event is registered as a navigation event and bb.event.payment-order.product-selected pub/sub event is registered to save the sender account in the widget.

CXP model

{
  "navigation": "bb.action.payment-order.select-product",
  "native": "SelectorWidget",
  "clients": "ProductSummaryClient",
  "packages": "com.backbase.android.client.productsummary",
  "events": "bb.event.payment-order.product-selected",
  "listViewKey": "0",
  "headers": "Products",
  "size": "10",
  "native.view": "DefaultSelectorWidgetView",
  "native.package": "com.backbase.android.widget.selector.views" //This preference is mandatory for your android implementation.
}

Step 2: After implementing the preferences above, call selectProduct method to load the SelectorWidget. This method will show the SelectorWidget and also pre-select the previously(if any existing) selected accounts.

Android

getContract().selectProduct();

iOS

self.contract?.selectProduct()

Step 3: Optionally bb.event.payment-order.product-selected pub/sub event can be registered in your view, in case you need to handle the result of the selection of the sender account. This pub/sub event is registered in the initiate by considering you select sender account during the creation of the payment order wrapper.

Select recipient account

Selecting a recipient follows exactly same steps with different navigation and event pub/sub events. On the other hand, considering that the user would like to select a contact from her/his contact list, we also register ContactsClient in the CXP model as follows:

Step 1:

CXP model

{
  "navigation": "bb.action.payment-order.select-recipient",
  "native": "SelectorWidget",
  "clients": "ProductSummaryClient, ContactsClient",

  "events": "bb.event.payment-order.recipient-selected, bb.event.payment-order.recipient-selected",

  "headers": "Products, Contacts",
  "size": "10",
  "native.view": "DefaultSelectorWidgetView",
  "native.package": "com.backbase.android.widget.selector.views" //This preference is mandatory for your android implementation.
}

Step 2:

Android

getContract().selectRecipient();

iOS

self.contract?.selectRecipient()

Save contact (only for SEPA payments)

To implement creation of a contact to the user’s own contact lists, you need use createContact method, this method is using ContactsClient to create a contact.

  1. Get recipient from the wrapper object, there is a recipient object that can be assigned when the result is taken from the Selector Widget during the assignment of the payment order recipient.

  2. This is an optional step, by considering that your "create contact view" is part of the validation view, and you submit the contact in another view then you need to pass shouldCreateContact boolean with the wrapper object from the validation view. If shouldCreateContact is set to true then you can proceed with the creating the contact.

  3. Submit the contact.

  4. Handle result of the contact creation, if it returns error response as null, that means the contact is created successfully.

Android

// Step 1
RecipientModel recipientModel = initiatePaymentOrderWrapper.getRecipientModel();
// Step 2
if(initiatePaymentOrderWrapper.shouldCreateContact() && recipientModel != null) {
    // Step 3
    getContract().createContact(recipientModel);
}

// Step 4
@Override
public void onContactCreationFinished(Response errorResponse) {
    if (errorResponse != null) {
        showSnackbar(R.string.failed_to_create_contact);
    }
}

iOS

// Step 1
guard let recipientModel = self.initiatePaymentOrderWrapper.recipientModel else { return }
// Step 2
if initiatePaymentOrderWrapper.shouldCreateContact {
    // Step 3
    self.contract?.createContact(with: recipientModel)
}

// Step 4
public func contactCreationFinished(_ error: Error?) {
    guard let error = error else {
        return
    }
    self.presentError(error: error)
}

Schedule a payment order

To schedule a payment use the PaymentOrderScheduleWidget sub-widget, and implement the PaymentOrderScheduleView contract to set up your view.

Step 1: Set up the necessary CXP preferences. See the example below:

CXP Model

{
  "navigation": "bb.action.payment-order.select-schedule",
  "native": "PaymentOrderScheduleWidget",
  "native.view": "NameOfYourCustomView",
  "native.package": "package.of.your.custom.view" //This preference is mandatory for your android implementation.
}

Step 2: Call either of these widget methods to calculate and adjust the Schedule or BBSchedule object with user inputs from the view.

Android & iOS

<contract>.selectSchedule() - publishes an event with the current schedule.

<contract>.getSchedule() - retrieves the state of the current schedule.

<contract>.getMinimumEndDate() - retrieves the minimum selectable end date.

<contract>.updateEndDate() - updates the schedule with a given end date and adjusts the number of payments based on the transfer frequency.

<contract>.updateStartDate() - updates the schedule with a given start date and adjusts the end based on the number of payments.

<contract>.updateFrequency() - updates the schedule with a given transfer frequency and end date in relation to the start date.

<contract>.updateNumberOfPayments() - updates the number of payments and adjusts the end date in relation to the start date and frequency.

<contract>.reset() - resets the schedule to default state.

@Override

}

public void updateViewWithSchedule(@Nullable Schedule schedule) {

}

    // Update view with changes to the end date
}

    // Update view with changes to the number of payments
}

    // Update view with pre-filled schedule information when a schedule is selected from the PaymentOrderWidget.
