All capabilities v3.5.0

This release note lists the changes to all capabilities of Widget Collection Retail Banking in this release.

Deprecations

The following features are deprecated from this release:

  • Support for these features ceases in v3.7.0.

    • As soon as Backbase Theme 2 is released, the next major version of Widget Collection Retail Banking will no longer support Backbase Theme 1.x.x. A migration path will become available with the release of Backbase Theme 2.

    • For iOS native widgets: BBBaseClient path property is deprecated in favor of BBDBSUrlPath initializer.

    • For Android native widgets: GenericListener is removed in favor of ResponseBodyListener. The new interfaces’s signature and usage the same, but the name makes it more clear what its purpose is.

Breaking changes

The following breaking changes have been introduced in this release:

  • For iOS native widgets:

    • Added missing annotations to nullability annotation to the entire iOS widget collection.

    • The URL extension is now provided by the native client collection, while previously the native clients only provided the version number and the endpoint URL extension: /v2/{end-point-url-extension}. For example, /v2/bill-pay/payments for the Bill Pay payments endpoint. From this release on, the native clients are providing the API, the version number and the endpoint postfix: {capability}-presentation-service/client-api/v2/{end-point-url-extension}. For example, billpay-presentation-service/client-api/v2/bill-pay/payments for the Bill Pay payments endpoint.

    • CommonUI was a support library for the default views. Now that default views have been removed in favor of sample views, CommonUI is no longer necessary and has thus been removed altogether, after a period of deprecation.

  • For Android native widgets:

    • The URL extension is now provided by the native client collection, while previously the native clients only provided the version number and the endpoint URL extension: /v2/{end-point-url-extension}. For example, /v2/bill-pay/payments for the Bill Pay payments endpoint. From this release on, the native clients are providing the API, the version number and the endpoint postfix: {capability}-presentation-service/client-api/v2/{end-point-url-extension}. For example, billpay-presentation-service/client-api/v2/bill-pay/payments for the Bill Pay payments endpoint.

    • The DBSDataImpl#buildGetRequest(URI, String, DBSDataProvider, Map) class is removed and replaced by the DBSDataImpl#buildGetRequest(URI, String, Params, DBSDataProvider) class.

    • CommonUI was a support library for the default views. Now that default views have been removed in favor of sample views, CommonUI is no longer necessary and has thus been removed altogether, after a period of deprecation.

    • Common Utils Kotlin is no longer released. All the Kotlin files are moved to common utils.

      • Two of the getAccounts methods are deprecated and replaced by their overloaded method.

    • For the iOS version of the Bill Pay Client:

      • The retrieveAccounts and retrieveAccountsWithParameters methods are deprecated and replaced by the retrieveAccountsWithCompletionHandler method.

    • For the Bill Pay Widget:

      • In the BillPayView interface, the following methods are deprecated and replaced:

Breaking changes

The following breaking changes have been introduced in this release:

  • For the Bill Pay Client, the following breaking changes apply:

    • In the Payment class, the paymentAccountNumber property is removed and replaced by the paymentAccount.paymentAccountNumber property.

    • In the Payment class, the paymentAccountNickName property is removed and replaced by the paymentAccount.paymentAccountNickName property.

    • The Payment class constructor is replaced with a new constructor containing these additional parameters: paymentAccount and payeeID.

    • The Payee class constructor is replaced with a new constructor containing this additional parameter: paymentServices.

    • The ElectronicPayee class constructor is replaced with a new constructor containing this additional parameter: paymentServices.

    • For iOS only:

      • The PayeeSearchResponse class has been removed from the Bill Pay Client, as it has been converted to a private class.

      • The searchPayeeByName method, which returns the PayeeSearchResponse object, has been removed from the Bill Pay Client. Instead use the searchPayeeByName method that returns a list of the PayeeSearch object.

  • For the iOS version of the Bill Pay Widget:

    • In sample code, the BillPayWidgetView class is removed.

Payments v3.5.0

New features

The new features in this release are:

  • New: Manage External Accounts For A2A Transfers (web)

    The new Connect external account widget allows the bank customer to connect an external account. Once the external account is connected, the bank customer can view it in the list of accounts along with its latest status.

  • Developers can send addition fields to the backend

    The InitiatePaymentOrder wrapper class is now extending from Additions class. When using the createInitiatePaymentOrder method from the wrapper class, it will also assign the additions property to the created InitiatePaymentOrder class instance.

    When developers use this feature, the backend also validates the additional fields of a payment order.

Self Enrollment v3.5.0

This release note explains the introduction of this new capability.

New features

The new features in this release are:

  • Introducing Self Enrollment

    Self Enrollment enables an existing bank customer to onboard onto the bank’s or financial institution’s digital platform in a fast and effortless way. Once a user has been successfully enrolled, they can easily log on to the digital platform with the newly created account credentials.

  • Customer identification

    This step confirms if the user is an existing customer by matching all their details against existing records within the financial institution.

  • Customer verification

    This step validates the identity of the user. It ensures that the user is the same person they say they are by matching their details against a third-party verification provider. For example, providing them out-of-wallet questions which only a specific user would know.

  • Setup username and password

    This step enables the user to create credentials to log on to the digital platform of the financial institution.

  • Enrollment success

    The Self Enrollment Confirmation widget prompts a successful enrollment message and enables bank customers to log in via the web digital banking app, or redirects them to download the mobile digital banking app.

  • Enrollment unsuccessful

    The Self Enrollment Locked Out widget notifies the user that they are now unable to continue in the enrollment process.

For further information please see Understand Self Enrollment.
