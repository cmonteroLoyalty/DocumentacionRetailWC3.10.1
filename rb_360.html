All capabilities v3.6.0

This release note lists the changes to all capabilities of Widget Collection Retail Banking in this release.

Deprecations

The following features are deprecated from this release:

  • Support for these features ceases in v3.7.0.

    • For iOS only:

      • In BBBaseClient, path component, and its initialisers are deprecated.

Breaking changes

The following breaking changes have been introduced in this release:

  • For Android only:

    • GenericListener is removed in favor of ResponseBodyListener. The signature and usage of the new interface are the same, but the name explains better its purpose.

  • Only triangular capabilities should be used, or all lean capabilities together with traditional ones should be used.

    The lean capabilities in Digital Banking Services are:

    • Contacts

    • Messages

    • Notifications

    • Payments

    The lean capability in Entitlements Services is:

    • Approvals

  • Microsoft Internet Explorer 11 is deprecated.

  • Microsoft Internet Explorer 10 is removed.

Account Statements v3.6.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • The Account Statement List widget migrated to Theme 2

    Theme 1 is still available with a feature flag.

  • Improved Account Statement List widget title

    The title of the widget was changed to View and download retail account statements to make its purpose clearer and to improve consistency among widget collections.

  • The statement category is no longer mandatory in the Account Statement List widget

Deprecations

This release includes the following deprecations:

  • In AccountStatementTableHeadComponent:

    • @Input() headers: Unused, headers were moved to the template.

    • @Output() sortingParamsChange: Output is redundant since the migration to Theme 2.

  • In AccountStatementTableComponent, bbAccountStatementTableCustomizable:

    • @Input() headers: Unused, headers were moved to the template.

    • @Input() formatsMap: Should be used from the constant.

    • setChangedSortingParams public method: The new onSort method should be used instead.

  • In AccountStatementTableRowComponent, bbAccountStatementTableRowCustomizable:

    • @Input() formatsMap: mimeTypeExtensions public property could be used instead.

    • Public property id: Unused since migration to Theme 2 templates.

    • Public method viewOnline: Unused since migration to Theme 2 templates.

    • Public method getDropdownHeaderId: Unused since migration to Theme 2 templates.

    • Public method getDownloadItemId: Unused since migration to Theme 2 templates.

  • In bbAccountStatementListCustomizable

    • Public property tableHeaders: Unused, headers were moved to the template.

    • Public property formatsMap: should be used from the constant.

  • The bbAccountStatementTableHeadCustomizable extension slot will be removed in v3.7.0 due to migration to the table component. Use bbAccountStatementTableCustomizable instead.

Bill Pay v3.6.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Improved widget titles

    Widgets have been renamed to make their purpose clearer, and to improve consistency among widget collections.

    The following web widgets have been retitled:

    • Bill Pay Payee Summary is now View bill payee summary.

    • Bill Pay Payments is now View bill payments.

    • Bill Pay Manage Payee is now Manage bill payee.

    • Bill Pay Manage Payments is now Manage bill payments.

    • Bill Pay Enrolment Guard is now Guard bill pay enrollment.

    • Bill Pay Auto Enrolment is now Enroll bill pay automatically.

    • Bill Pay Ebills Payveris Enrolment is now Enroll Payveris eBills.

    • Bill Pay Payee List is now View bill payees.

    The following native mobile widgets have been retitled:

    • Bill Pay is now Manage bill payments.

    • Bill Pay Form is now Create bill payment.

    • Bill Pay Schedule is now Schedule bill payment.

  • Manage bill payments now has a Cancel button

    A Cancel button has been added to the Manage bill payments widget’s form and preview pages.

  • View bill payees now uses a notification routing service

    This routing service enables overlapping notifications to route the View bill payees widget.

  • Error message improvements

    • The error message text now references support instead of system administrator. This aligns Bill Pay with the other Retail Banking widgets. This change affects the following Bill Pay widgets:

      • Enroll bill pay automatically

      • Manage bill pay

      • Manage bill payments

      • View bill payees

      • View bill payments

    • A Manage bill payments error message title has been changed from Form is not loading to the more meaningful Payment information is not loading.

  • Create bill payment has been updated to use bb.action.bill-pay.edit-payment instead of the deprecated bb.action.bill-pay.payment-edit.

Bug fixes

This release includes bug fixes for the following widgets:

  • Enroll bill pay automatically

    • OBPAY-2319: In some error scenarios the loading indicator failed to change to the error state.

  • Manage bill pay

    • OBPAY-1940: The successful payee notification buttons now navigate to their respective widgets.

  • Manage bill payments

    • OBPAY-2320: The successful payee notification button now navigates to the payments list.

    • OBPAY-2367: Payment and schedule details are now not shown until a payee is selected.

  • View bill payee summary

    • OBPAY-2231: The Notification internationalisation name was incorrectly referring to View bill payees instead of View bill payee summary

  • Enroll Payveris eBills

    • OBPAY-2280: The password visibility toggle for the password field has been fixed.

Breaking changes

This release includes the following breaking change:

  • The date format has been changed from dateTime to date-only, for example from 2018-06-21T00:00:00.000-04:00 to 2018-06-21. Use the date-only format for the following fields:

    • earliestPaymentDate

    • nextPaymentDate

    • paymentDate

    This affects the Bill Pay Form and Bill Pay native mobile widgets, and the Bill Pay client.

  • Native mobile: Methods are removed from the Messages client and Manage messages widget

      • The toggleConversationReadStateByRootMessageId:completionHandler and toggleConversationReadStateByRootMessageId:requestId:completionHandler methods are removed in the native client. Use toggleConversationsReadStateWithRequest:completionHandler method instead.

MessagesDataHttpModule.forRoot({

})

Due to the move to the lean service architecture, the following interfaces were updated:

  • DraftDraft interface is replaced with PostDraft

  • ConversationDraftsPostResponse interface is replaced with CreateDraftResponse

Payments v3.6.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Retrieve the external account

    You can now retrieve an external account through Payveris with the native Payment Order client.

  • Widget Migration to Theme 2.0 (web) The following widgets have been migrated to Backbase Theme 2.0

    • Initiate Payment

    • Upcoming Payments

    • Stop Checks

    • Stop Checks List

  • Initiate omni payment widget (web)

    The Initiate omni payment widget is a base widget that is fully customizable with a TypeScript configuration. The Initiate omni payment widget always requires a developer to either configure an out-of-the-box payment type, or to create and configure a custom payment type.

    This widget ships with out of the box configurations for the following payment types:

    • Domestic wire (US)

    • ACH credit transfer (US)

    For more information, see:

    • Adopt the Initiate omni payment widget

Breaking changes

This release includes the following breaking changes:

  • Removed the following properties in the native Payment Order client and Payment Order widget:

    • debtor from BBValidatedPaymentOrder

    • debtorAccount from BBInitiatePaymentOrder and BBInitiatePaymentOrderWrapper

    • debtorAccount in BBValidatedPaymentOrder

    • creditTransferTransactionInformation from BBValidatedPaymentOrder

    • creditTransferTransactionInformation from BBInitiatePaymentOrder and BBInitiatePaymentOrderWrapper

    • creditor from BBInitiateTransaction and BBIdentifiedTransaction

    • initializers in BBInitiatePaymentOrder and BBInitiateTransaction

    • creditorAccount in BBInitiateTransaction and BBIdentifiedTransaction

    • creditorBank in BBInitiateTransaction and BBIdentifiedTransaction

  • Removed the following classes in the native Payment Order client and Payment Order widget:

    • BBIdentifiedCreditTransaction

    • BBInitiateCreditTransaction

    • BBDebtorAccount

    • BBCreditorAccount

    • BBInitiateCreditorAccount

  • The following deprecated properties have been removed from the Initiate Payment and Upcoming Payments widgets:

    Removed

    Replacement

    debtor/debtorAccount

    originator/originatorAccount

    creditTransferTransactionInformation

    transferTransactionInformation

    creditTransferTransactionInformation.creditor

    transferTransactionInformation.counterparty

    creditTransferTransactionInformation.creditorAccount

    transferTransactionInformation.counterpartyAccount

    creditTransferTransactionInformation.creditorBank

    transferTransactionInformation.counterpartyBank

Bug fixes

This release includes the following bug fixes:

  • MAINT-8797: Payment Order Client - Expose DateTime fields as String

    Previously, the date-time fields in the Payment Order client were not formatted as a raw string. Now you can use the date-time fields as a raw string format to conveniently convert such fields to your local timezone.

  • MAINT-9093: Upcoming Payments (web) - Default payment params can’t be overriden

    When extending the Upcoming Payments widget it was not possible to change the default payment params because they were provided at component level instead of the module. This has been fixed.

  • MAINT-9095: Upcoming Payments (web) - the spinner spins forever when there is an error

    The error handling for the widget has been fixed to show an appropriate error state.

  • MAINT-9275: Initiate Payments (web) - error from the /validate endpoint is ignored

    If an error was received when the widget sent a request to the endpoint POST //payment-order-presentation-spec/client-api/v2/payment-orders/validate to validate the payment payload, the error was ignored. This has been fixed, and the user flow is stopped if validation fails.

Known issues

This release has the following known issues:

  • Discovered in v3.6.0

Known issues

This release has the following known issues:

  • Discovered in v3.6.0

    • MAINT-9720: Cannot have two omni-payment widgets with different payment types.

      When two or more widgets customize the Initiate omni-payment widget and use the withConfig method to provide the payment configuration, the second or last configuration made overrides all the previous configurations.

      To work around this issue, provide the paymentConfig and the business function as inputs to the Initiate omni-payment widget component, in the wrapper widget’s template. Do not use the withConfig method. This issue is fixed with the 3.6.2 release.

Transactions v3.6.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Native mobile: Ability to filter transactions by incoming and outgoing transactions

    The native mobile Transactions widget now enables users to find specific transactions within their transactions history based on the ability to filter incoming and outgoing transactions.

  • Native mobile: Ability to view the merchant name and icon of a transaction

    The native mobile Transactions widget now provides the ability to display the merchant icon and name of a transaction so that bank customers can easily identify their transactions.

  • Native mobile: Ability to have a more comprehensive visual indication of transactions with a new set of icons

    The native mobile Transactions widget now provides the ability to display different icons for transactions based on available data. In addition to category icons, now transactions can display merchant icons, transaction type icons, incoming transaction icon and outgoing transaction icon. Which icon to be displayed depends on the following priority order: merchant icons > category icons > transaction type icons > Incoming and outgoing transaction icons. For example, if the merchant information is not present for a transaction then category information is checked and an icon for category is shown, if it exists. If category information is also not present different icons are shown based on the transaction type. Finally, if transaciton type information is also not present, then either an incoming or outgoing transaction icon is shown.

  • Native mobile: Filter transactions widget

    The Filter transactions widget enables users to find specific transactions within their transactions history based on several filters criteria, such as incoming and outgoing transaction and so on.

  • Web: Improved widget titles

    Widgets have been renamed to make their purpose clearer, and to improve consistency among widget collections.

    The following web widget has been renamed:

    • Transactions List is now View transactions list.

  • Web: Widget migration to Theme 2.0

    The View transactions list widget has been migrated to Backbase Theme 2.0.

Bug fixes

This release includes the following bug fixes:

  • MAINT-7757: Android Transaction Widget JSON de/serialization

    TransactionsWidget created multiple objects of Gson which in turn created multiple other objects on which it depended.

  • MAINT-7650: TransactionWidget’s onTransactionSelected() function causes memory issue

    The onTransactionSelected function created a Gson object every time when a transaction was selected.

Deprecations

This release includes the following deprecations:

Support for these features ceases in v3.7.0.

  • Native mobile Transactions widget:

    The following deprecations in the native mobile widgets are due to the introduction of the new Product client that replaces the Product summary client. It is advised to start using the new product property and method once you migrate your code to the Products client.

    • Android:

      • The KEY_PRODUCT preference is deprecated and replaced by the KEY_SELECTED_PRODUCT preference.

      • The getProduct method is deprecated and replaced by the getSelectedProduct method, which now returns an instance of the BaseProduct class from the new Products client.

      • The updateViewWithProduct(Product product) method is deprecated and replaced by the updateViewWithProduct method.

    • iOS:

      • The product property is deprecated and replaced by the selectedProduct property.

      • The loadTransactions method is deprecated and replaced by the updateViewWithProduct method.
